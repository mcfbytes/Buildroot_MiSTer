# ADR 0002 — Keep `/etc/resolv.conf` a symlink, using Buildroot's default (answers Q2)

**Status:** Accepted (2026-07-12) — decided by @mcfbytes
**Supersedes:** `docs/phase0-review.md` Q2; PLAN.md ledger #3a, invariant A8
**Impact:** P2.3 (init/config parity); **PLAN §3 text must be corrected**

## Decision

Do nothing bespoke. Ship Buildroot's stock skeleton, which already provides:

```
/etc/resolv.conf -> ../run/resolv.conf     (work/buildroot/system/skeleton/etc/resolv.conf)
```

Stock points at `../tmp/resolv.conf` instead. Both are symlinks into a tmpfs;
we take Buildroot's, per the standing principle of preferring an upstream
default over code we maintain.

## Rationale

Verified safe to retarget `/tmp` → `/run`:

- Nothing in the stock rootfs `/etc` or in Main_MiSTer references
  `/tmp/resolv.conf` **by path** — the only reference is the symlink itself.
- `dhcpcd` writes `/etc/resolv.conf` and follows the symlink wherever it points
  (live box: `# Generated by dhcpcd from wlan0.dhcp`).
- Downloader's restore target is `/etc/resolv.conf`, not the symlink's
  destination (`Downloader_MiSTer/src/downloader/constants.py:101`:
  `('/media/fat/linux/resolv.conf', '/etc/resolv.conf')`).
- Both `/tmp` and `/run` are tmpfs in stock `fstab`.

## Correction to an earlier hypothesis — do not repeat it

It is tempting to argue the symlink is *forced* by a read-only rootfs (bootargs
say `ro`). **That is false.** The live device mounts root read-**write**:

```
/dev/loop8 on / type ext4 (rw,noatime,nodiratime)
```

The symlink is a design choice, not a necessity. Recorded because the false
version of this reasoning is load-bearing if believed — it would imply a regular
file breaks DNS, which it would not.

## Consequences

**The `resolv.conf` user-file restore remains a silent no-op.** Downloader's
`copy()` does `open(full_target, 'wb')`, which *follows* the symlink and writes
into `/run/resolv.conf` (was: `/tmp/resolv.conf`) **inside the offline image**,
which the tmpfs then shadows at boot. A user who customises `resolv.conf` loses it
on every update.

This is stock behaviour and has never worked on any MiSTer. We are keeping parity,
**deliberately** — not because the bug is absent.

Note this is not really *our* bug: the defect is in Downloader following a symlink
during restore. Worth reporting upstream (see ADR 0003 for the general "engage
upstream at publish time" posture).

**PLAN.md must be corrected.** Invariant A8 — "all six user-file-restore
destinations are plain regular files" — is factually wrong, and PLAN §3's
"no symlink-into-tmpfs schemes" rule is contradicted by the design we are now
deliberately adopting. Both need rewording, not just a footnote.
