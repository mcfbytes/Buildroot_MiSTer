### ⚠ Finding: not every `S`-script is an inline script

- `/etc/init.d/S45bluetooth` is a **symlink** to `/bin/bluetoothd`, not a regular file.

`rcS`/`rcK` (verbatim below) run every `/etc/init.d/S??*` entry with
`$i start` / `$i stop` regardless of whether it's a real file or a
symlink (their only guard is `[ ! -f "$i" ]`, which follows symlinks).
So `S45bluetooth start` really execs `/bin/bluetoothd start` --
verified below, `/bin/bluetoothd` is itself a conventional
start/stop/restart/reload/renew/hcireset control script (same shape
as the other services), just placed at `/bin/bluetoothd` instead of
written inline under `/etc/init.d/`. It execs the real daemon at
`/usr/libexec/bluetooth/bluetoothd` with args `-n -E -C`, and --
directly relevant to P2.4's writable-paths audit and P3.5's Bluetooth
parity task -- **this is stock's actual BT-pairing persistence
mechanism**: on `start` it creates (if missing) and loop-mounts a
64KiB×32=2MiB ext4 image at `/media/fat/linux/bluetooth` onto
`/var/lib/bluetooth` (`sync,dirsync,nodiratime,noatime`), i.e. pairing
state lives in a small dedicated image file on the FAT data partition,
not in tmpfs and not directly in the (read-only) rootfs. A `renew`
action deletes that image to reset all pairings. Full verbatim text
of `/bin/bluetoothd` is in `etc-init-scripts-full.txt` alongside this
doc (listed there under its resolved path since it isn't literally
under `/etc/init.d/`).

`/etc/shadow` is deliberately **not** reproduced verbatim in this repo
(it contains real crypt() password hashes baked into the public stock
image; there is no reason to also re-publish them verbatim in a new
repo). Structural summary instead -- for each account, whether the shadow
field is a login-capable hash or a locked/no-login marker (`*`/`!`):

| Account | Shadow field | Login-capable? |
|---|---|---|
| `root` | (hash present, not reproduced here) | **yes** |
| `daemon` | `*` | no (locked) |
| `bin` | `*` | no (locked) |
| `sys` | `*` | no (locked) |
| `sync` | `*` | no (locked) |
| `mail` | `*` | no (locked) |
| `www-data` | `*` | no (locked) |
| `operator` | `*` | no (locked) |
| `nobody` | `*` | no (locked) |

| `dhcpcd` | `*` | no (locked) |
| `sshd` | `*` | no (locked) |
| `ftp` | `*` | no (locked) |

**Finding:** `root` is the only account with a login-capable password hash
(a SHA-512 crypt hash, `$5$...`); every service account is locked (`*`).
Combined with `sshd_config`'s `PermitRootLogin yes` (verbatim above) and
`proftpd.conf` (verbatim above, no explicit `RootLogin off`), stock ships
**remote root login enabled by default, gated only by a fixed, publicly
shipped default password**. This project must not silently change that
(parity requirement, P3.7) but must document the risk prominently in the
user-facing FAQ (P4.8) rather than treating it as a secret. The specific
plaintext default password is widely published in the existing MiSTer
community documentation/wiki; it is intentionally not restated here --
this doc's job is to confirm *that* a fixed default credential + remote
root login exists, which is the fact P3.7/P4.8 need, not to be another
place that publishes it.

## A8 user-file-restore destinations: regular-file check

Evidence method: `os.path.islink()`/`os.path.isfile()` on the extracted
rootfs tree (symlinks are preserved as real symlinks by the `debugfs
rdump` extraction method used throughout this inventory -- verified
docs/reference-materials.md section 3), cross-checked directly against
the raw ext4 image with `debugfs -R "stat <path>"` for the one finding
below.

| Destination | Type | Regular file (A8 requirement)? |
|---|---|---|
| `/etc/hostname` | regular file | yes |
| `/etc/hosts` | regular file | yes |
| `/etc/network/interfaces` | regular file | yes |
| `/etc/resolv.conf` | symlink -> `../tmp/resolv.conf` | **NO** |
| `/etc/dhcpcd.conf` | regular file | yes |
| `/etc/fstab` | regular file | yes |

### ⚠ Finding: `/etc/resolv.conf` is a symlink, not a regular file

`/etc/resolv.conf -> ../tmp/resolv.conf` in stock (`/tmp` is tmpfs per
`fstab`, verbatim above). This contradicts the documented A8 invariant
that all six user-file-restore destinations are plain regular files.
Traced through `Downloader_MiSTer`'s actual restore code
(`src/downloader/file_system.py`, the `copy()` method used by
`linux_updater.py`'s `_restore_user_files()`): it does
`open(full_target, 'wb')`, which **follows** a symlink rather than
replacing it. Run against the *offline-mounted, unbooted* new image
(where `/tmp` is just an ordinary empty ext4 directory, not a live
tmpfs), the restore therefore silently writes the user's custom
`resolv.conf` content to `/tmp/resolv.conf` **inside the persisted
image** -- not to a stable `/etc` location. On the next boot, the
`tmpfs` mount over `/tmp` shadows that file immediately, so the
restored content is invisible/lost from the moment the system boots.
In practice this is likely harmless *by original design* --
`/etc/resolv.conf` being a tmpfs symlink exists so `dhcpcd` can
rewrite DNS servers on a read-only root, and a custom static
`resolv.conf` is probably rare/moot for DHCP-managed installs -- but
it means the "restore custom resolv.conf" step of the updater
contract is a **no-op in stock**, not the regular-file copy A8 and
P2.3 assume. P2.3 must either preserve this exact stock behavior
(symlink to tmpfs, silently-discarded restore) or explicitly diverge
and document why (e.g. make `/etc/resolv.conf` a real regular file so
the restore actually sticks, accepting that a read-only root then
needs a different writable-DNS mechanism, P2.4's problem to solve).
The other five destinations are genuine regular files and the
documented restore mechanism works for them as described.
