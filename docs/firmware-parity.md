# Firmware parity: `/lib/firmware` vs. stock (P3.3)

This is the P3.3 deliverable for the **firmware population** half of
"Module loading & firmware infra" (TASKS.md). The **module-autoload** half
(kmod/depmod/eudev/xz-compression) was already done — see the P3.3 (core)
commit and `configs/mister_de10nano_defconfig`'s `BR2_PACKAGE_HOST_KMOD_XZ` /
`BR2_PACKAGE_KMOD_TOOLS` lines. This document covers only `/lib/firmware`.

**Target:** `docs/stock-inventory/firmware.md` — the authoritative 66-file
stock inventory (a 67th stock file, `xow_dongle.bin`, is P3.2's
`xow-firmware` package and is not repeated here).

**Method:** every file was checked two ways, not assumed from either source
alone, and the result below is **build-verified** — cross-checked against
the actual built `output/images/rootfs.tar`, not just `output/target/` or
Config.in reasoning:

1. Which Buildroot `linux-firmware` (or `wireless-regdb`) Config.in
   sub-option, if any, installs that literal path — read directly from
   `work/buildroot/package/linux-firmware/linux-firmware.mk` (1054 lines,
   read in full) and `work/buildroot/package/linux-firmware/Config.in`.
   This step alone **undersells** actual coverage: linux-firmware's own
   install step runs a WHENCE-driven pass that adds symlinks for files not
   explicitly selected but aliased (in upstream's own WHENCE) to a file
   that *was* installed — several stock files turned out to be reproduced
   this way despite no sub-option naming them (see below).
2. For every file **not otherwise reproduced**, whether *this project's
   pinned 6.18.33 kernel source* (`output/build/linux-6.18.33/`, not
   upstream HEAD, not stock's kernel) has an in-tree driver that requests
   that exact firmware path via `request_firmware()`/`MODULE_FIRMWARE()` —
   checked by grepping the actual built kernel source tree, cited per-file
   below — and whether the file exists at all in the pinned
   `linux-firmware-20251011.tar.xz` (checked via `tar tf` on the actual
   downloaded, hash-verified archive, not assumed).
3. `tar -tvf output/images/rootfs.tar | grep lib/firmware` against the
   full 66-file stock list, diffed with `comm`.

non-inventory siblings that rode along with an enabled sub-option (e.g.
`RTL_87XX_BT` also installs `rtl_bt/rtl8723cs_cg_fw.bin`, which stock never
shipped).

**A note on symlinks vs. regular files:** stock's inventory counts 66
*regular* files (`find -type f`). Several of our reproduced files land as
**symlinks**, not regular files — e.g. top-level `mt7601u.bin` →
`mediatek/mt7601u.bin`, `mt7650.bin` → `mediatek/mt7650.bin`,
`mt7662.bin`/`mt7662_rom_patch.bin` likewise, and (see below)
`rtl_bt/rtl8723d_config.bin` → `rtl8821c_config.bin`,
`rtlwifi/rtl8192eefw.bin` → `rtl8192eu_nic.bin`,
`rtlwifi/rtl8723bs_{ap_wowlan,nic,wowlan}.bin` → the corresponding
`rtl8723bu_*.bin`. This is **upstream linux-firmware's own WHENCE-driven
install behavior** (`copy-firmware.sh`'s logic, reproduced by
`work/buildroot/package/linux-firmware/linux-firmware.mk`'s
`LINUX_FIRMWARE_INSTALL_FW`), not something this project introduced, and it
is functionally identical to a regular file — `request_firmware()` resolves
symlinks transparently at the VFS level. Confirmed via
`tar -tvf output/images/rootfs.tar`, e.g.:

```
lrwxrwxrwx  ./usr/lib/firmware/rtl_bt/rtl8723d_config.bin -> rtl8821c_config.bin
lrwxrwxrwx  ./usr/lib/firmware/rtlwifi/rtl8192eefw.bin -> rtl8192eu_nic.bin
lrwxrwxrwx  ./usr/lib/firmware/rtlwifi/rtl8723bs_ap_wowlan.bin -> rtl8723bu_ap_wowlan.bin
```

**This corrected two Config.in-only conclusions reached before the build**:
`rtl_bt/rtl8723d_config.bin` and `rtlwifi/rtl8192eefw.bin` initially looked
like gaps (no sub-option's `LINUX_FIRMWARE_FILES` list names them
literally), but upstream's own WHENCE aliases both to a sibling file that
*is* installed by an enabled sub-option, so they show up in the image
without needing `linux-firmware-extra` at all. Same for the three
`rtlwifi/rtl8723bs_*.bin` files (aliased to their `rtl8723bu_*` USB
siblings) — present in the image even though no in-tree `rtl8723bs` (SDIO)
driver exists to use them (harmless unused bytes, not a gap).

## `package/linux-firmware-extra` (the gap-fill package)

Four files land in no Buildroot `linux-firmware` Config.in sub-option
despite being genuinely present in upstream linux-firmware **and** having a
confirmed in-tree 6.18.33 kernel consumer:

| File | In-tree consumer |
|---|---|
| `mediatek/mt7610u.bin` | `drivers/net/wireless/mediatek/mt76/mt76x0/mt76x0.h: MT7610U_FIRMWARE "mediatek/mt7610u.bin"` |
| `mediatek/mt7622pr2h.bin` | `drivers/bluetooth/btmtk.h: FIRMWARE_MT7622 "mediatek/mt7622pr2h.bin"` |
| `mediatek/mt7668pr2h.bin` | `drivers/bluetooth/btmtk.h: FIRMWARE_MT7668 "mediatek/mt7668pr2h.bin"` |
| `rtlwifi/rtl8723befw_36.bin` | `drivers/net/wireless/realtek/rtlwifi/rtl8723be/sw.c: fw_name = "rtlwifi/rtl8723befw_36.bin"` |

Rather than patching `work/buildroot/package/linux-firmware/linux-firmware.mk`
directly (that tree is gitignored/untracked — `work/` — so a hand-edit there
would not survive a `work/` re-extraction and is not part of this project's
source of truth), `package/linux-firmware-extra` is a small satellite
Buildroot package that:

- downloads the **exact same** `linux-firmware-20251011.tar.xz`, from the
  same `$(BR2_KERNEL_MIRROR)/linux/kernel/firmware` site, hash-pinned with
  the **same published sha256** the sibling `linux-firmware` package's own
  `.hash` file carries (not a new/foreign source — verified identical);
- extracts only the four needed members (plus `WHENCE` and three
  `LICENCE.*` files, for `legal-info`) via a custom `EXTRACT_CMDS` that
  accounts for the tarball's `linux-firmware-<version>/` wrapper directory
  (`--strip-components=1`, member names given wrapper-prefixed — this bit
  the first build attempt, see "Build issues hit" below), avoiding a
  second full unpack of the ~580 MiB upstream tree;
- installs them under their stock-matching paths in
  `$(TARGET_DIR)/lib/firmware/`.

**Three further candidates were tried and dropped** from this package:
`brcm/BCM20702A1-0b05-17cb.hcd`, `rtl_bt/rtl8723d_config.bin`,
`rtlwifi/rtl8192eefw.bin`. The latter two turned out not to need a satellite
package at all (present via WHENCE symlink, see above). The first —
`brcm/BCM20702A1-0b05-17cb.hcd` — is a genuine gap with **no available
source**: see "Flagged" below.

## Flagged: no confirmed source (3 files, not fabricated)

Per the task's explicit rule — do not fabricate a source — these are
documented, not sourced from an unpinned/unverified mirror:

- **`brcm/BCM20702A1-0b05-17cb.hcd`** (Broadcom BT patch for an ASUS-branded
  BCM20702A1 adapter, VID:PID `0b05:17cb`). `drivers/bluetooth/btbcm.c` (in
  kernel 6.18.33) constructs firmware paths as `brcm/BCM%s%s.hcd` from a
  chip's VID:PID, so the driver *could* use this file if present — but the
  pinned `linux-firmware-20251011.tar.xz` was checked directly
  (`tar tf ... | grep '\.hcd'`) and contains **exactly one** `.hcd` file in
  the entire tarball: `brcm/BCM-0bb4-0306.hcd` (HTC's VID, an unrelated
  device). No Broadcom-Bluetooth Config.in menu exists in Buildroot's
  `linux-firmware` at all (only Broadcom *WiFi* — `BRCM_BCM43XX` etc. — has
  toggles), and upstream itself does not carry this VID:PID's blob in the
  pinned snapshot. Broadcom BT patches are vendor/device-specific and
  linux-firmware only carries contributed subsets — this one was
  apparently never contributed upstream.

- **`rt2870_sw_ch_offload.bin`** — not in upstream linux-firmware's
  `WHENCE` (fetched and searched; only the ordinary `rt2870.bin` entry
  exists, which *is* shipped), and no in-tree 6.18.33 driver requests it by
  that literal name (`rt2800usb.h`'s only firmware macro is
  `FIRMWARE_RT2870 "rt2870.bin"`). Public information describes it as an
  alternate/optional firmware variant from third-party vendor tooling, not
  part of the kernel.org linux-firmware project.

**Recommendation for review:** both are candidates for "genuinely
vestigial in stock, no driver ever loaded them by that literal name" —
omitting them costs nothing functionally. If a known-good source exists for
either, provide it and it can be hash-pinned into
`package/linux-firmware-extra` the same way the other four gap files are.

- **`RTL8192E/{boot.img,data.img,main.img}`** is listed separately below
  (obsolete, not "no source" — the driver was deleted from Linux) but is
  worth noting it is *also* absent from the pinned upstream tarball
  (`tar tf ... | grep -i 'rtl8192e/'` → zero hits) — upstream pruned the
  firmware alongside the driver, so there is no path to reproducing it even
  if desired.

## Justified omissions — obsolete (6 files)

Driver removed from the kernel, or firmware removed from upstream
linux-firmware, or both — confirmed against this project's actual pinned
6.18.33 kernel source and the actual pinned linux-firmware tarball, not
assumed:

- **`RTL8192E/boot.img`, `RTL8192E/data.img`, `RTL8192E/main.img`** —
  consumed only by `drivers/staging/rtl8192e` (the PCIe RTL8192E driver),
  which was **deleted from the kernel in Linux 6.13** ("staging: rtl8192e:
  delete the driver"). Confirmed directly: `output/build/linux-6.18.33/
  drivers/staging/` has no `rtl8192e*` entry, and
  `grep -rl "RTL8192E/" output/build/linux-6.18.33/drivers` returns zero
  hits. Also absent from the pinned linux-firmware tarball itself (see
  above) — upstream removed the firmware too.
- **`mediatek/mt7662u.bin`, `mediatek/mt7662u_rom_patch.bin`** — superseded/
  renamed, not missing: the in-tree `mt76x2u` USB driver (present, built)
  requests the **top-level** `mt7662.bin`/`mt7662_rom_patch.bin` (already
  shipped, confirmed via `usb_mcu.c: request_firmware(&fw, MT7662_FIRMWARE
  /MT7662_ROM_PATCH, ...)`), not `mediatek/mt7662u*.bin`. That stock
  filename is the old out-of-tree driver's naming convention.
- **`rtl_bt/rtl8192ee_fw.bin`, `rtl_bt/rtl8192eu_fw.bin`** — these DO exist
  in the pinned linux-firmware tarball (confirmed via `tar tf`), but have
  **zero in-tree consumer**: `grep -n "8192" drivers/bluetooth/btrtl.c`
  returns nothing — the in-tree Realtek BT chip-ID table has no
  RTL8192EE/EU entry in 6.18.33 at all (unlike `rtlwifi/rtl8192eefw.bin`
  below, where the *WiFi* driver source still exists, just Kconfig-gated
  off, this is a case where no code path in the kernel could ever request
  this filename). Not added to `linux-firmware-extra`: shipping firmware
  with no possible consumer anywhere in the kernel tree isn't parity, it's
  dead weight.

## Justified omission — no in-tree consumer, would need a new driver package (1 file)

- **`rtlwifi/rtl8723defw.bin`** — exists in the pinned linux-firmware
  tarball (confirmed via `tar tf`), but there is no `rtl8723de` directory
  anywhere under `drivers/net/wireless/realtek/rtlwifi/` in this kernel
  (only `rtl8192d`, `rtl8192de`, `rtl8192du`, `rtl8723ae`, `rtl8723be`,
  `rtl8723com` exist) — mainline never carried an in-tree RTL8723DE driver.
  Reproducing this file's *usability* would need a new out-of-tree driver
  package (P3.1-scope, not P3.3). `rtlwifi/rtl8723bs_{ap_wowlan,nic,
  wowlan}.bin` have the identical "no in-tree SDIO driver" situation but
  are present in the image anyway (WHENCE-aliased to their `rtl8723bu_*`
  USB siblings which a real sub-option installs) — harmless unused bytes,
  not something this package needed to add deliberately.

## A note on `rtlwifi/rtl8192eefw.bin` and `CONFIG_RTL8192EE`

`rtlwifi/rtl8192eefw.bin` **is present** in the built image (WHENCE symlink
to `rtl8192eu_nic.bin`, see above) — but its actual in-tree driver source
(`drivers/net/wireless/realtek/rtlwifi/rtl8192ee/`) is not currently built:
`board/mister/de10nano/linux.config` has no `CONFIG_RTL8192EE` line at all
(only `CONFIG_RTL8192CU=m` among the rtlwifi family, per P3.1). This is a
**P3.1/kernel-config gap, not a P3.3 firmware gap** — flagging it for the
orchestrator rather than fixing it here, since kernel-config changes are a
full-rebuild-risk decision (per the project's build lessons) out of this
task's remit. The firmware being present already means no follow-up would
be needed if that config gap is ever closed.

## Build issues hit and fixed

1. **First `make all` attempt failed** (`tar: ... Not found in archive` for
   all seven originally-planned `linux-firmware-extra` members). Root
   cause: the upstream tarball has a `linux-firmware-20251011/` wrapper
   directory around every member (confirmed via `tar tf`), same as why the
   sibling `linux-firmware` package's own `EXTRACT_CMDS` pipes through
   `tar --strip-components=1`. The custom `EXTRACT_CMDS` here didn't
   account for it. Fixed: member names now given wrapper-prefixed
   (`$(addprefix linux-firmware-$(LINUX_FIRMWARE_EXTRA_VERSION)/, ...)`)
   with `--strip-components=1` on the extraction.
2. **After that fix, 3 of the original 7 members still failed** (`brcm/
   BCM20702A1-0b05-17cb.hcd`, `rtl_bt/rtl8723d_config.bin`,
   `rtlwifi/rtl8192eefw.bin`) — genuinely absent from the pinned tarball
   (verified via its own file listing), not an extraction bug. Two of the
   three turned out not to be needed anyway (WHENCE symlinks from an
   already-enabled sub-option supply them); the third
   (`brcm/BCM20702A1-0b05-17cb.hcd`) is flagged above. Dropped all three
   from `linux-firmware-extra`'s member list; second rebuild succeeded
   (`make linux-firmware-extra-dirclean && make all`, exit 0).

## Size impact

See `docs/size-budget.md` — `/lib/firmware` totals **3.1 MiB** in the built
image (68 regular files + 23 symlinks); P3.3's own addition (beyond P3.2's
already-committed xow-firmware) is **≈ 2.9 MiB**. The image remains at
**60.6% free** (310.7 MiB / 512 MiB), well above the 15% floor.
